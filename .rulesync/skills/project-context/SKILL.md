# 项目上下文速览

被调用时，按下面结构产出一份简洁、可复用的项目上下文摘要（不要逐行罗列代码）：

1. **项目目标**：这个项目解决什么问题，当前阶段/里程碑
2. **架构概览**：核心模块、数据流、关键边界（一张文字版依赖图即可）
3. **关键约束**：性能/兼容/合规/不可改动的部分
4. **共享约定**：命名、错误处理、提交规范（指向 `.rulesync/rules/`）
5. **验证方式**：怎么本地验证一个改动是对的（lint / test / 手动步骤命令）

要求：
- 信息来源优先级：仓库内 README / docs / 根规则 > 代码推断；推断的部分明确标注"推断"
- 控制在 ~300 字；这是给后续任务复用的"地图"，不是百科
- 如果关键信息缺失，明确列出"未知，需向维护者确认"，不要编
