# 结构化调试(diagnose)

> 内核(对齐全局 CLAUDE.md ④ Goal-Driven):**只有可执行的验证才真正约束调试**。
> 先把"这个 bug 出现了没有"变成一条机器能跑、确定、快的 pass/fail 信号,你才算真正在排查;否则只是在猜。
> 别一上来就改代码——按下面六步走,每步先想清楚再动手(CLAUDE.md ①)。

## 阶段 1 — 建反馈回路(这一步就是整个 skill)

目标:一条 **快、确定、agent 能自己跑** 的 pass/fail 信号,精确对应这个 bug。
有了它,你几乎一定能定位根因;没有它,后面全是空中楼阁。

可选的构造方式(挑最快能搭出来的):
- 一个当前会失败的测试(最理想)
- 一条复现命令 + 对输出的断言(grep 退出码 / 比对快照)
- 一个最小复现脚本(把无关依赖全砍掉)
- 差分回路:对"好"与"坏"两个输入/版本跑同一命令,比对差异
- 性能问题:固定输入下的耗时/内存阈值断言

把回路往 **更快、更准、更确定** 的方向打磨——偶发 bug 先想办法把它逼成稳定复现(固定随机种子、并发降到 1、放大循环次数等)。

## 阶段 2 — 复现

确认这条回路触发的是 **用户报告的那个失败**,而不是旁边一个长得像的 bug;且复现率高到能边改边看。复现不了就先回到阶段 1 加压,不要往下走。

## 阶段 3 — 提假设(先列再验,别边改边猜)

写下 **3–5 条排好序、可证伪** 的假设,每条都长这样:

> 「若 <X> 是根因,则 <改动 Y> 会让 bug 消失。」

排序:先验"改动最小 / 最可能 / 一发命中就能区分一大片"的。没有可证伪形式的假设不算假设。

## 阶段 4 — 埋点验证

每个探针对准一条具体预测,一次只验一条,验完就知道该假设真/假。
- 优先用调试器 / 断点看真实值,其次才是打日志
- 调试日志一律加 **唯一前缀**(如 `DIAG-xxx:`),方便阶段 6 一把清干净
- 不要为了"顺便看看"撒一堆无关日志

## 阶段 5 — 修复 + 回归测试

- **先写回归测试,再改代码**:测试要架在 **正确的接口/调用缝(seam)** 上,复现的是 bug 在真实调用点的样子,而不是贴着实现写
- 改动只碰根因(CLAUDE.md ③ Surgical):别顺手重构、别清无关代码
- 跑阶段 1 的回路 + 新回归测试,都绿了才算修好

## 阶段 6 — 清理 + 复盘

- 删掉所有 `DIAG-` 临时埋点和复现脚本(除非已转成正式测试)
- 确认 **用户最初那个场景** 不再复现
- 在 commit message 里记一句:根因是哪条假设、为什么——给下一个踩坑的人留路标(别写进代码注释)

## 自检清单

```
[ ] 有一条 agent 能自己跑的 pass/fail 回路,且对准的是真 bug
[ ] 假设是可证伪的,验证前就排好了序
[ ] 回归测试架在正确的 seam 上,会因为这个 bug 失败
[ ] 改动每一行都能追溯到根因,无顺手改动
[ ] 临时埋点已清,原始场景已确认不复现
```

## 何时不用

明显的拼写/类型/编译错误、一眼能看出的笔误——直接改,别套流程。本 skill 是给"看不出原因、复现费劲、或性能回归"这类难缠问题用的。
